# បញ្ហា និងដំណោះស្រាយ

## (Problems and Solutions in the Development of KneaYerng Service Center App)

ក្នុងអំឡុងពេលនៃការអភិវឌ្ឍកម្មវិធី KneaYerng Service Center ដែលជាប្រព័ន្ធគ្រប់គ្រងការលក់ផលិតផល និងសេវាជួសជុលទូរស័ព្ទ ក្រុមការងារបានជួបប្រទះនូវបញ្ហាបច្ចេកទេសជាច្រើនទាំងផ្នែក Mobile Application (Flutter) និងផ្នែក Backend (Laravel)។ បញ្ហានីមួយៗត្រូវបានវិភាគរកមូលហេតុដើម (Root Cause) ហើយដោះស្រាយដោយវិធីសាស្ត្រសមស្របទៅតាមលក្ខណៈនៃប្រព័ន្ធ។ ខាងក្រោមនេះជាបញ្ហាសំខាន់ៗ ដែលបានជួបប្រទះ ព្រមទាំងដំណោះស្រាយជាក់ស្តែងដែលបានអនុវត្ត។

## ១. បញ្ហាប្រព័ន្ធកម្មង់ទិញដាច់ដោយខ្លោចទាំងស្រុង ពេល Firebase មិនបានកំណត់រចនាសម្ព័ន្ធ

**បញ្ហា៖** ប្រព័ន្ធកម្មង់ទិញ (Order) ត្រូវការផ្ញើការជូនដំណឹង (Push Notification) ទៅអតិថិជនតាមរយៈ Firebase Cloud Messaging រាល់ពេលដែលមានកម្មង់ទិញថ្មីកើតឡើង ឬមានការប្តូរស្ថានភាពកម្មង់ទិញ។ ប៉ុន្តែនៅពេលដែលឯកសារសម្ងាត់ (Credentials) របស់ Firebase មិនទាន់ត្រូវបានដាក់នៅលើ Server ផលិតកម្ម (Production) ទេនោះ ខណៈដែល Constructor របស់ Service ការជូនដំណឹង (FirebasePushNotificationService) ព្យាយាមចូលប្រើ Credentials នោះ វាបានបង្កឱ្យមាន Exception ដែលគ្មានការគ្រប់គ្រង (Unhandled Exception) ហើយដោយសារ OrderResource ត្រូវការប្រើ Service តាមដានកម្មង់ទិញ (OrderTrackingService) ដែលអាស្រ័យលើ Service ការជូនដំណឹងនេះ ទាល់តែម្តង ការបង្កើតកម្មង់ទិញគ្រប់ប្រភេទ (ទទួលនៅហាង បញ្ជូនដល់ផ្ទះ ប្រាក់សុទ្ធ ឬ KHQR) សុទ្ធតែបរាជ័យជាមួយកំហុស 500 Internal Server Error ដែលជាកំហុសធម្មតា មិនបញ្ជាក់ច្បាស់ពីមូលហេតុ ធ្វើឱ្យអតិថិជនមិនអាចបញ្ជាទិញបានសោះ។

**ដំណោះស្រាយ៖** ក្រុមការងារបានកែប្រែ Service ការជូនដំណឹង ឱ្យព័ទ្ធជុំវិញការចែងផ្នែកចំាប់ផ្តើម (Factory Initialization) ដោយ try/catch ដើម្បីកត់ត្រាការព្រមាន (Log Warning) ហើយបិទមុខងារជូនដំណឹង Push នៅពេលគ្មាន Credentials ជំនួសឱ្យការគាំង។ ដូចគ្នានេះផងដែរ ផ្នែកដែលប្រើ Service តាមដានកម្មង់ទិញនៅក្នុង OrderResource ក៏ត្រូវបានព័ទ្ធជុំវិញ try/catch ដែរ ដើម្បីធានាថា API នឹងតែងតែត្រឡប់លទ្ធផលកម្មង់ទិញឱ្យបាន ទោះបីជា Firebase មិនទាន់ត្រូវបានកំណត់រចនាសម្ព័ន្ធក៏ដោយ។ វិធីសាស្ត្រនេះហៅថា Graceful Degradation គឺប្រព័ន្ធនៅតែបន្តដំណើរការមុខងារសំខាន់ៗបាន ទោះបីជាមុខងារបន្ទាប់បន្សំខ្លះមិនអាចប្រើបានក៏ដោយ ជំនួសឱ្យការគាំងប្រព័ន្ធទាំងស្រុង។

## ២. បញ្ហាភាពស្មុគស្មាញនៃការទូទាត់ប្រាក់តាម KHQR និងការទទួលទំនិញដោយ QR Code

**បញ្ហា៖** ដើម្បីឱ្យសមស្របទៅនឹងបរិបទកម្ពុជា ប្រព័ន្ធត្រូវការគាំទ្រការទូទាត់ប្រាក់តាម Bakong KHQR ដែលជា Payment Gateway ស្តង់ដារជាតិ បន្ថែមលើការទូទាត់ប្រាក់សុទ្ធ និង ABA។ ការអនុវត្តកូដ KHQR ដំបូងមាន Logic ច្របូកច្របល់ និងបាក់ខ្ចាត់ខ្ចាយនៅក្នុង KhqrPaymentController មួយឯកសារតែមួយ ដែលធ្វើឱ្យពិបាកថែទាំ និងពិបាករកមូលហេតុ ពេលមានកំហុសក្នុងការបង្កើត QR Code ឬការផ្ទៀងផ្ទាត់ស្ថានភាពទូទាត់។ ជាមួយគ្នានោះ មុខងារទទួលទំនិញនៅហាង (Pickup) ក៏ត្រូវការ QR Code ដោយឡែកសម្រាប់សម្គាល់សំបុត្រទទួលទំនិញ (Pickup Ticket) ប៉ុន្តែគ្មានទម្រង់រួម (Unified Format) ច្បាស់លាស់រវាង QR ទូទាត់ប្រាក់ និង QR ទទួលទំនិញនោះទេ ដែលបង្កការច្របូកច្របល់ក្នុងការអភិវឌ្ឍ Flutter App ផ្នែកអេក្រង់ Cart និង Checkout។

**ដំណោះស្រាយ៖** ក្រុមការងារបានញែក Logic ការទូទាត់ប្រាក់ចេញជា Service ដោយឡែកគឺ KhPayService ដែលធ្វើជា Layer កណ្តាលសម្រាប់ទាក់ទងជាមួយ Bakong Open API (BakongOpenApiService) រួមជាមួយ PaymentController ថ្មីសម្រាប់គ្រប់គ្រងលំហូរទូទាត់ប្រាក់ឱ្យដាច់ដោយឡែកពី Logic កម្មង់ទិញ។ សម្រាប់ផ្នែកទទួលទំនិញ ប្រព័ន្ធបានបន្ថែមជួរឈរ QR Short Code ដោយឡែកនៅក្នុងតារាង Orders (តាមរយៈ Migration) និងពង្រឹង PickupTicketService ឱ្យបង្កើត និងផ្ទៀងផ្ទាត់សំបុត្រទទួលទំនិញដោយប្រើ Code ខ្លីដែលងាយស្រួលសម្រាប់បុគ្គលិកស្កេន ដាច់ដោយឡែកពី QR ទូទាត់ប្រាក់ទាំងស្រុង។ ការញែក Service ឱ្យមានទំនួលខុសត្រូវច្បាស់លាស់ម្នាក់មួយ (Single Responsibility) នេះ បានជួយកាត់បន្ថយកំហុស និងធ្វើឱ្យកូដងាយស្រួលតេស្ត និងថែទាំជាងមុន។

## ៣. បញ្ហាការប្រើប្រាស់ Splash Screen ធ្វើឱ្យពេលចំាប់ផ្តើមកម្មវិធីយឺត

**បញ្ហា៖** ការរចនាដំបូងបាននិពន្ធ SplashScreen ដោយឡែកមួយ ដែលបង្ខំឱ្យកម្មវិធីរង់ចាំរយៈពេលថេរ (kSplashDuration) មុននឹងបង្ហាញអេក្រង់ Onboarding ឬអេក្រង់ដើម ដែលធ្វើឱ្យអ្នកប្រើប្រាស់មានអារម្មណ៍ថាកម្មវិធីចំាប់ផ្តើមយឺត ទាំងដែលជាក់ស្តែងទិន្នន័យចំាបាច់ (ដូចជា Token ការចូលប្រើប្រាស់ និងភាសា) អាចផ្ទុកបានលឿនជាងពេលរង់ចាំថេរនោះទៅទៀត។

**ដំណោះស្រាយ៖** ក្រុមការងារបានដកការរង់ចាំថេរ (Future.delayed) ចេញ ហើយលុបឯកសារ splash_screen.dart ចេញពីគម្រោងទាំងស្រុង ដោយប្តូរទៅប្រើ CircularProgressIndicator សាមញ្ញនៅក្នុង _StartupGate ដែលជា Widget តែមួយធ្វើទាំងការផ្ទុកទិន្នន័យចំាបាច់ និងសម្រេចចំណាត់ការនាំទិសទៅអេក្រង់សមស្រប (Onboarding ឬ Main Navigation) ភ្លាមៗពេលទិន្នន័យរួចរាល់ ដោយមិនចំាបាច់រង់ចាំរយៈពេលថេរទេ។ វិធីនេះធ្វើឱ្យពេលចំាប់ផ្តើមកម្មវិធីលឿនជាងមុន ហើយធ្វើឱ្យលំហូរនាំទិស (Navigation Flow) មានភាពច្បាស់លាស់ និងសាមញ្ញជាងមុន។

## ៤. បញ្ហាការផ្ញើ OTP តាមរយៈអ្នកផ្តល់សេវាច្រើនប្រភេទ

**បញ្ហា៖** ដើម្បីធានាបាននូវភាពជឿទុកចិត្តក្នុងការផ្ញើកូដ OTP សម្រាប់ផ្ទៀងផ្ទាត់គណនី ប្រព័ន្ធបានសាកល្បងភ្ជាប់ជាមួយអ្នកផ្តល់សេវា SMS ច្រើនប្រភេទ ដូចជា Unimatrix, Infobip និង Twilio។ ការមាន Provider ច្រើនបែបនេះ ធ្វើឱ្យកូដ OTP Service ដើមមានលក្ខណៈស្មុគស្មាញ ព្រោះត្រូវសរសេរ Logic ផ្សេងគ្នាសម្រាប់ Provider នីមួយៗ ហើយប្រសិនបើ Provider ណាមួយផ្លាស់ប្តូរ API ឬមានបញ្ហាបណ្តាញ វានឹងប៉ះពាល់ដល់លំហូរការចុះឈ្មោះ (Register) និងផ្ទៀងផ្ទាត់អត្តសញ្ញាណរបស់អតិថិជនទាំងអស់ ប្រសិនបើគ្មានយន្តការគ្រប់គ្រងកំហុសត្រឹមត្រូវ។

**ដំណោះស្រាយ៖** ប្រព័ន្ធបានបង្កើត OtpDeliveryService ជា Layer កណ្តាលដែលជ្រើសរើស Provider ដោយផ្អែកលើការកំណត់រចនាសម្ព័ន្ធ (config('services.otp.sms_provider')) ដោយ Unimatrix ត្រូវបានកំណត់ជា Provider ស្រេចចិត្ត (Default) រីឯ Infobip និង Twilio ត្រូវបានរក្សាទុកជា Option បន្ថែម។ រាល់ការផ្ញើតាម Infobip ត្រូវបានព័ទ្ធជុំវិញដោយ try/catch ដើម្បីកត់ត្រាកំហុស ហើយត្រឡប់ false ដោយមិនធ្វើឱ្យកម្មវិធីគាំង ប្រសិនបើការផ្ញើបរាជ័យ។ ការរចនាបែបនេះអនុញ្ញាតឱ្យក្រុមការងារផ្លាស់ប្តូរ Provider សម្រាប់ផលិតកម្មបានយ៉ាងងាយ ដោយគ្រាន់តែប្តូរ Environment Variable តែមួយ ដោយមិនចំាបាច់កែកូដ ហើយក៏ជួយឱ្យប្រព័ន្ធមានភាពធន់ទ្រាំ (Resilience) ខ្ពស់ ប្រសិនបើ Provider មួយណាមួយប្រឈមនឹងបញ្ហា។

## ៥. បញ្ហាខ្នាតធំនៃអេក្រង់ (Large Screen Complexity) ធ្វើឱ្យពិបាកថែទាំ

**បញ្ហា៖** ដោយសារតម្រូវការមុខងារកើនឡើងជាបន្តបន្ទាប់ អេក្រង់សំខាន់ៗជាច្រើននៅក្នុង Flutter App ដូចជា checkout_flow_screen.dart (ដល់ទៅ ៤,០៦៥ បន្ទាត់), product_detail_screen.dart (២,៨៨៥ បន្ទាត់), home_screen.dart (២,៥៣៦ បន្ទាត់), profile_screen.dart, search_results_screen.dart, cart_screen.dart, និងអេក្រង់ផ្សេងទៀតសរុបជិត ១៤ File មានចំនួនបន្ទាត់កូដច្រើនជាង ១,០០០ បន្ទាត់ ដោយផ្គុំផ្សំទាំង UI, State Management, និង Business Logic នៅក្នុង File តែមួយ ដែលធ្វើឱ្យការអាន ការតេស្ត និងការកែប្រែប្រឈមនឹងហានិភ័យខ្ពស់ក្នុងការបង្កកំហុសថ្មី (Regression) រាល់ពេលដែលត្រូវកែប្រែ។

**ដំណោះស្រាយ៖** ក្រុមការងារបានកាត់បំបែកគ្រប់អេក្រង់ដែលមានបន្ទាត់កូដច្រើនជាង ១,០០០ ដោយផ្លាស់ផ្ទីផ្នែក UI ដែលអាចញែកដាច់ដោយឡែក (ដូចជា Card, Section, Step, Dialog, Bottom Sheet) ឱ្យទៅជា Widget File ដោយឡែកនៅក្នុង Folder widgets/ ដែលនៅជាប់នឹងអេក្រង់ដើមនីមួយៗ (ឧ. lib/screens/cart/widgets/, lib/screens/products/widgets/) ខណៈដែល Business Logic ទាំងអស់ (ការហៅ API, Navigation, State Management) នៅតែស្ថិតនៅក្នុង File អេក្រង់ដើម។ លទ្ធផលគឺបន្ទាត់កូដក្នុង File អេក្រង់ដើមថយចុះយ៉ាងសំខាន់ ឧទាហរណ៍ checkout_flow_screen.dart ពី ៤,០៦៥ ទៅ ៩៩៤ បន្ទាត់ product_detail_screen.dart ពី ២,៨៨៥ ទៅ ៧៤៩ បន្ទាត់ និង home_screen.dart ពី ២,៥៣៦ ទៅ ៥៦៨ បន្ទាត់ ដោយគ្មានការផ្លាស់ប្តូរលក្ខណៈដំណើរការ (Behavior) ឡើយ ត្រូវបានផ្ទៀងផ្ទាត់រាល់ជំហានដោយ flutter analyze ដែលឆ្លងផុតដោយគ្មានកំហុស។ បន្ថែមលើនេះ Widget ដែលអាចប្រើឡើងវិញបាននៅទូទាំងគម្រោង ដូចជា circle_back_button.dart និង empty_state_view.dart ក៏ត្រូវបានបង្កើតឡើង ដើម្បីកាត់បន្ថយការចម្លងកូដដូចគ្នាគ្នាឡើងវិញ។ ដោយឡែក checkout_flow_screen.dart ដែលជា Logic ស្មុគស្មាញបំផុតរបស់កម្មវិធី នៅតែសល់ប្រមាណ ៩៩៤ បន្ទាត់ដែលជា Business Logic សុទ្ធ។ សម្រាប់ដំណាក់កាលអនាគត គម្រោងគ្រោងនឹងប្តូរទៅប្រើ State Management Library ដូចជា Riverpod ឬ Bloc ជំនួសការប្រើ setState សាមញ្ញ ដើម្បីឱ្យកូដមាន Structure កាន់តែច្បាស់ និងងាយធ្វើតេស្តដោយស្វ័យប្រវត្តិ។

## ៦. បញ្ហាការប្រែប្រួលនៃលក្ខណៈតួអក្សរចុងបន្ទាត់ (Line Ending) ក្នុងការងារជាក្រុម

**បញ្ហា៖** ដោយសារក្រុមការងារប្រើប្រព័ន្ធប្រតិបត្តិការខុសគ្នា (Windows និង Mac/Linux) ឯកសារកូដមួយចំនួនមានលក្ខណៈតួអក្សរចុងបន្ទាត់ខុសគ្នា (LF និង CRLF) ដែលធ្វើឱ្យ Git កត់សម្គាល់ការផ្លាស់ប្តូរលើបន្ទាត់ទាំងមូលនៃឯកសារ ទាំងដែលជាក់ស្តែងគ្មានការផ្លាស់ប្តូរខ្លឹមសារពិតប្រាកដឡើយ បង្កការលំបាកក្នុងការត្រួតពិនិត្យកូដ (Code Review) និងបង្កការប៉ះទង្គិច (Merge Conflict) ញឹកញាប់។

**ដំណោះស្រាយ៖** ដំណោះស្រាយដែលត្រូវអនុវត្តគឺការបង្កើតឯកសារ .gitattributes នៅគល់គម្រោង Flutter ដើម្បីបង្ខំឱ្យ Git ប្រើលក្ខណៈតួអក្សរ LF ជាស្តង់ដារសម្រាប់ឯកសារកូដគ្រប់ប្រភេទ ដោយមិនគិតពីប្រព័ន្ធប្រតិបត្តិការរបស់សមាជិកក្រុមនីមួយៗ។ វិធីនេះធានាបាននូវភាពស្របគ្នានៃប្រវត្តិកូដ (Git History) ហើយជួយឱ្យការផ្តល់យោបល់លើ Pull Request មានភាពច្បាស់លាស់ ដោយឃើញតែខ្លឹមសារពិតប្រាកដដែលត្រូវបានកែប្រែ។

## សេចក្តីសន្និដ្ឋាន

តាមរយៈការអភិវឌ្ឍប្រព័ន្ធ KneaYerng Service Center App ក្រុមការងារបានជួបនិងបញ្ហាបច្ចេកទេសជាច្រើនប្រភេទ ចាប់ពីបញ្ហាការកំណត់រចនាសម្ព័ន្ធ (Firebase Credentials) បញ្ហាស្ថាប័នកូដ (Code Architecture) រហូតដល់បញ្ហាការងារជាក្រុម (Line Ending)។ ការដោះស្រាយបញ្ហានីមួយៗបានផ្អែកលើគោលការណ៍សំខាន់ៗចំនួនពីរគឺ ទីមួយ ការគ្រប់គ្រងកំហុសឱ្យបានសុក្រឹត (Graceful Error Handling) ដើម្បីកុំឱ្យបញ្ហាមួយផ្នែកប៉ះពាល់ដល់ប្រព័ន្ធទាំងមូល និងទីពីរ ការញែកទំនួលខុសត្រូវកូដឱ្យបានច្បាស់លាស់ (Separation of Concerns) តាមរយៈការប្រើ Service Layer ដែលជួយឱ្យកូដមានភាពងាយស្រួលថែទាំ និងពង្រីកមុខងារថែមទៀតនៅពេលអនាគត។
